# Karpathy 指南

[English](../karpathy-guidelines/SKILL.md) | 简体中文

减少常见 LLM 编码错误的行为指南，源自 [Andrej Karpathy 对 LLM 编码陷阱的观察](https://x.com/karpathy/status/2015883857489522876)。

**权衡：** 这些指南偏向谨慎优先，速度其次。对于琐碎任务，请自行判断。

## 1. 编码前思考

**不要假设。不要掩盖困惑。说明权衡。**

实现前：
- 明确写出你的假设。不确定时，先询问。
- 如果存在多种理解，列出它们，避免默默选择。
- 如果存在更简单的方案，说明理由。该提出异议时要提出。
- 如果有不清楚的地方，先停下。指出困惑点并询问。

## 2. 简洁优先

**用最少代码解决问题。不做额外推测。**

- 不添加需求之外的功能。
- 不为一次性代码创建抽象。
- 不添加未被要求的“灵活性”或“可配置性”。
- 不为不会发生的场景增加错误处理。
- 如果写了 200 行且 50 行能解决，请重写。

问自己：“资深工程师会觉得这过于复杂吗？” 如果会，请简化。

## 3. 精准修改

**只修改必须修改的内容。只清理自己造成的问题。**

编辑现有代码时：
- 不“改进”相邻代码、注释或格式。
- 不重构正常工作的代码。
- 匹配现有风格，即使你更倾向于其他写法。
- 如果发现无关的死代码，只提及，不删除。

当你的改动产生孤立代码时：
- 删除因你的改动而变得无用的导入、变量和函数。
- 不删除预先存在的死代码，除非用户明确要求。

检验标准：每一行改动都应能直接追溯到用户请求。

## 4. 目标驱动执行

**定义成功标准。循环验证直到达成。**

将任务转化为可验证目标：
- “添加验证” → “为无效输入编写测试，然后让测试通过”
- “修复 bug” → “编写能复现 bug 的测试，然后让测试通过”
- “重构 X” → “确保重构前后测试都通过”

对于多步骤任务，给出简短计划：
```
1. [步骤] → 验证：[检查]
2. [步骤] → 验证：[检查]
3. [步骤] → 验证：[检查]
```

强成功标准让你可以独立循环推进。弱标准（“让它能用”）会带来持续澄清。
